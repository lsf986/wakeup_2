# Loona 无唤醒词语音唤醒需求文档

## 1. 文档信息

- 模块名称：无唤醒词语音唤醒模块
- 所属产品：Loona Deskmate App / Loona Deskmate 设备侧能力
- 文档类型：需求整理稿
- 来源文档：
  - 《语音唤醒场景case与分级.pdf》
  - 《语音交互需求-20260430.pdf》
- 目标读者：产品、设计、客户端、设备侧、算法、测试
- 关联模块：注意力模块、环境感知模块、状态管理模块、多模态交互模块

## 2. 目标

本阶段只实现从持续监听状态到正确判断用户是否在对 Loona 说话，并在判断成立时触发唤醒。

具体目标：

1. Loona 持续接收环境中的声音和多模态感知信号。
2. 系统结合朝向、视线、唇动、声源、距离等信息，判断用户是否正在对 Loona 说话。
3. 当判断用户正在对 Loona 说话时，系统触发唤醒事件。
4. 当用户不是在对 Loona 说话时，系统保持未唤醒状态，避免误唤醒。
5. 本阶段不依赖固定唤醒词，不把说出“Loona”或其他唤醒词作为触发前提。

## 3. 当前范围

### 3.1 本阶段需要实现

- 持续监听状态下的用户说话检测。
- 多模态判断用户是否在对 Loona 说话。
- 无唤醒词唤醒意图识别。
- 有效唤醒事件输出。
- 误唤醒控制。
- 唤不醒控制。
- P0 / P1 / P2 场景分级与测试用例。

### 3.2 本阶段不需要实现

- 固定唤醒词唤醒。
- 唤醒成立后的后续语音交互链路。
- Loona 输出过程中的用户再次开口处理。
- Agent 任务规划、任务抢占、排队和决策逻辑。
- 语义理解层面的复杂意图判断。

## 4. 核心定义

### 4.1 持续监听状态

Loona 在待机状态下持续接收音频和多模态感知信号，用于判断是否存在用户对 Loona 发起说话的行为。

### 4.2 无唤醒词语音唤醒

用户未说固定唤醒词，但系统通过多模态信息判断用户正在对 Loona 说话，并触发唤醒事件。

### 4.3 有效唤醒事件

系统判断“用户正在对 Loona 说话”后输出的事件。该事件只表示唤醒判断成立，不包含后续语音交互或任务执行逻辑。

### 4.4 误唤醒

用户并未对 Loona 说话，但系统错误触发唤醒事件。

### 4.5 唤不醒

用户明确正在对 Loona 说话，但系统未触发唤醒事件。

## 5. 输入信号

系统可使用以下信号进行综合判断：

1. 声音内容：是否存在人声、是否为短音、是否接近自然发话。
2. 声源位置：声音是否来自用户所在方向。
3. 声源距离：声音是否来自 Loona 附近的有效交互距离。
4. 用户头部朝向：用户是否面向或转向 Loona。
5. 用户视线方向：用户是否看向 Loona，或说完后看向 Loona 等待响应。
6. 唇动：用户是否存在与声音同步的唇部动作。
7. 当前注意力目标：Loona 当前关注的用户是否为说话人。
8. 环境噪声：是否存在远处人声、电视、电脑外放、播客、直播等背景声。
9. 场景状态：单人办公、多人讨论、用户移动、用户工作中等上下文。

## 6. 输出结果

系统输出应至少包含：

1. 是否触发唤醒：`wakeup = true / false`。
2. 判断置信度：用于后续调试和阈值优化。
3. 触发依据：例如朝向、视线、唇动、声源方向、距离等主要命中因素。
4. 目标用户标识：多人场景下尽量标识触发唤醒的目标用户。
5. 失败或拒绝原因：例如背景音、非目标用户、人际对话、无意义短音、距离过远等。

## 7. 功能需求

### 7.1 持续监听

1. 系统应在 Loona 待机状态下持续监听音频信号。
2. 系统应同步接收朝向、视线、唇动、声源位置、距离等多模态信号。
3. 系统不应依赖固定唤醒词作为唤醒入口。
4. 系统应能够区分近处用户发话与远处背景声。

### 7.2 多模态唤醒判断

1. 系统应综合多个信号判断用户是否在对 Loona 说话。
2. 当用户面向 Loona 发话，且声源、距离、唇动等信号一致时，应触发唤醒。
3. 当用户起始面向 Loona，但说话过程中轻微转头或视线偏移时，仍应判断为有效唤醒。
4. 当用户先看电脑，随后边说边转向 Loona，且整体意图明确时，应尽量判断为有效唤醒。
5. 多人场景中，系统应优先判断当前注意力目标是否在对 Loona 说话。
6. 系统不应仅凭单一信号触发唤醒，例如只检测到人声、只检测到看向 Loona、只检测到唇动。

### 7.3 误唤醒控制

1. 多人自然聊天时，即使有人无意朝向 Loona，也不应触发唤醒。
2. 用户明显在对他人说话时，即使中途看向 Loona，也不应触发唤醒。
3. 远处人声、电视播报、电脑外放视频、播客、直播等背景音不应触发唤醒。
4. 用户看向 Loona 但只是发呆、思考或自言自语时，不应触发唤醒。
5. “嗯”“啊”“诶”“啧”等无意义短音不应单独触发唤醒。
6. 系统不应把“用户出声”本身等同于对 Loona 说话。

### 7.4 唤不醒控制

1. 单人办公场景中，用户明确面向 Loona 发话时，系统必须触发唤醒。
2. 用户在近距离面向 Loona 说短句时，系统必须触发唤醒。
3. 多人场景中，当前注意力目标明确面对 Loona 发话时，系统必须触发唤醒。
4. 用户边做事边说话，但过程中存在明确面向 Loona 的动作时，系统应尽量触发唤醒。
5. 用户未全程面向 Loona，但最后有明显等待 Loona 响应的动作时，系统应尽量触发唤醒。

## 8. 场景分级标准

### 8.1 P0 必须实现

满足以下任一特征时，归为 P0：

- 单人核心办公场景。
- 用户对 Loona 说话的意图非常明确。
- 属于基础可用性的底线。
- 做不到会直接影响主体验。

### 8.2 P1 理应实现

满足以下特征时，归为 P1：

- 高频真实场景。
- 信号存在一定复杂性，但仍属于正常使用。
- 用户会合理期待系统做到。

### 8.3 P2 可放宽标准

满足以下特征时，归为 P2：

- 弱朝向、弱声量、弱意图。
- 多因素叠加导致识别难度明显更高。
- 当前版本可适当放宽，不作为核心上线阻塞项。

## 9. 应唤醒场景

### 9.1 P0 必须实现

#### Case 1：面向 Loona 开始说话，过程中持续面向

- 场景：用户正对 Loona，自然开始说话，整个说话过程中朝向稳定且明确。
- 示例：
  - “今天天气怎么样？”
  - “帮我看一下这个。”
  - “继续发邮件。”
- 预期：
  - 系统应判断用户正在对 Loona 说话。
  - 系统应触发唤醒事件。
  - 用户能明确感知 Loona 进入被唤醒状态。

#### Case 2：面向 Loona 开始说话，过程中允许轻微转头或视线偏移

- 场景：用户起始时明确面向 Loona，但说话过程中自然低头、看屏幕或轻微偏头。
- 示例：
  - “你帮我总结一下……我看下……对，就是这段。”
  - “这个邮件什么意思？”
- 预期：
  - 系统应判断为有效对 Loona 发话。
  - 不应因中途短暂未面向 Loona 而取消唤醒判断。

#### Case 3：单人办公场景中，面向 Loona 说短句

- 场景：用户在桌前办公，转向 Loona 说出极短命令或问题。
- 示例：
  - “知道了。”
  - “看这个。”
  - “什么意思？”
  - “讲重点。”
  - “发吧。”
- 预期：
  - 系统应触发唤醒事件。
  - 不应因为句子短而漏唤醒。

#### Case 4：多人场景中，当前注意力目标明确面对 Loona 说话

- 场景：多人在场，当前用户已处于 Loona 注意力范围内，并明确朝向 Loona 发起输入。
- 示例：
  - “你觉得刚才他说的重点是什么？”
  - “帮我记一下这个结论。”
  - “你再重复一遍。”
- 预期：
  - 系统应判断当前目标用户正在对 Loona 说话。
  - 系统应触发唤醒事件。
  - 不应因背景中还有其他人说话而漏掉目标用户。

### 9.2 P1 理应实现

#### Case 5：边做事边对 Loona 说话，过程中存在明确面向

- 场景：用户在操作电脑、整理桌面、拿东西等同时，对 Loona 发起语音输入。
- 示例：
  - “你等一下帮我总结这个文件。”
  - “我这两天日程排得满吗？”
  - “帮我记住这个。”
- 预期：
  - 系统应较稳定地判断为有效对 Loona 发话。
  - 不应要求用户全程静止、全程正对 Loona。

#### Case 6：先看电脑，随后边说边转向 Loona

- 场景：用户原本在看电脑，开口后逐步将注意力和朝向转向 Loona。
- 示例：
  - “这个地方什么意思啊，你帮我解释一下。”
  - “这个邮件谁发的啊，你帮我总结一下。”
- 预期：
  - 系统理应判断为有效对 Loona 发话。
  - 不应因前半句未完全朝向 Loona 而整体漏唤醒。

#### Case 7：多人场景中，未全程面向 Loona，但有明确面向动作

- 场景：多人一起讨论时，用户并非全程正对 Loona，但语气和行为明显是在向 Loona 发问。
- 示例：
  - “你帮我记一下刚才这个结论……”
  - “这个会后帮我整理成三点。”
- 预期：
  - 系统理应触发唤醒事件。
  - 在多人场景中仍应能抓住明确的对话意图。

#### Case 8：多人场景中，未全程面向 Loona，但最后有等待响应动作

- 场景：多人一起讨论时，用户并非全程正对 Loona，但说完后明显注视 Loona，等待回复。
- 示例：
  - “这个问题你怎么看？”
- 预期：
  - 系统理应触发唤醒事件。
  - 等待响应动作可作为重要辅助信号。

### 9.3 P2 可放宽标准

#### Case 9：完全不面向 Loona，或仅短暂瞥向 Loona，但确实在对 Loona 说话

- 场景：用户始终看着电脑或别处，没有明显朝向 Loona，但语言内容和语气是在对 Loona 发起输入。
- 示例：
  - “你帮我把这段总结一下。”
  - “这个是什么意思？”
  - “帮我看看这个逻辑。”
- 预期：
  - 系统可争取识别成功。
  - 当前版本可放宽，不作为必须稳定实现的场景。

#### Case 10：用户在移动过程中对 Loona 说话

- 场景：用户起身、走动、转身、收拾东西时，顺口对 Loona 说一句话。
- 示例：
  - “待会提醒我开会。”
  - “你继续，我拿个东西。”
- 预期：
  - 系统可争取识别成功。
  - 当前版本可放宽标准。

#### Case 11：多人场景下，弱朝向、弱声量地对 Loona 说话

- 场景：多人同时在场，目标用户既没有明显朝向，也不是特别大声，但确实想对 Loona 说话。
- 示例：
  - “稍后提醒我。”
- 预期：
  - 系统可争取识别。
  - 当前版本可适当放宽成功率要求。

## 10. 不应唤醒场景

### 10.1 P0 必须实现

#### Case 1：多人对话中，用户彼此自然交流，过程中有无意朝向

- 场景：多人自然聊天，过程中存在无意朝向，但没人对 Loona 发起输入。
- 示例：
  - “这个方案明天再看吧。”
  - “你先把文档发我一下。”
- 预期：
  - 系统不应触发唤醒事件。
  - 不应把多人自然交流误判为对 Loona 说话。

#### Case 2：用户明显在对他人说话，过程中有无意朝向

- 场景：用户自然转头与旁边的人交流，Loona 在场，但不是对话对象。
- 示例：
  - “你刚才说的是这个意思吗？”
  - “等我一下，我马上来。”
- 预期：
  - 系统不应触发唤醒事件。
  - 不应错误抢入人与人之间的对话。

#### Case 3：远处人声 / 背景音频 / 电视播报

- 场景：Loona 周围存在远处人声、视频音频、环境播报等持续环境音。
- 示例：
  - 远处有人聊天。
  - 电脑外放视频对白。
  - 电视新闻播报。
  - 播客、直播、人声节目。
- 预期：
  - 系统不应触发唤醒事件。
  - 不应被背景音高频带起。

### 10.2 P1 理应实现

#### Case 4：用户看向 Loona，但并未真正发起交互

- 场景：用户看了 Loona 一眼，但只是发呆、思考、顺口说话，并非真正对 Loona 说话。
- 示例：
  - “嗯……”
  - “啊，烦死了。”
  - “这个有点麻烦。”
- 预期：
  - 系统理应不触发唤醒事件。
  - 不应仅因“看向 + 出声”就触发。

#### Case 5：用户边工作边发出无意义短音

- 场景：用户办公过程中发出“嗯”“啊”“诶”等短音，但没有明确交互意图。
- 示例：
  - “嗯。”
  - “啊。”
  - “诶？”
  - “啧。”
- 预期：
  - 系统理应不触发唤醒事件。
  - 不应把无意义短音当成对话开始。

### 10.3 P2 可放宽标准

#### Case 6：用户自言自语或读屏幕内容

- 场景：用户一边工作一边念内容、复述文案、轻声自语，过程中有不持续但多次的无意面向，但并不是对 Loona 说话。
- 示例：
  - “这个标题是不是要改一下……”
  - “嗯，这段写得不太顺。”
  - “第一部分、第二部分、第三部分……”
- 预期：
  - 系统不应默认触发唤醒事件。
  - 不能把“用户出声”本身当成对 Loona 说话。

## 11. 重点问题

### 11.1 误唤醒

典型来源：

- 远处人声。
- 背景对话。
- 电视或电脑外放音频。
- 非交互对象发言。
- 用户无意义短音。
- 用户看向 Loona 但没有真实交互意图。

目标：误唤醒不应影响核心办公体验，不应让 Loona 频繁进入错误唤醒状态。

### 11.2 唤不醒

典型来源：

- 用户已明确面向 Loona 说话，但系统没有触发唤醒。
- 用户短句输入太短，被系统忽略。
- 多人场景中目标用户被背景人声掩盖。
- 用户说话开头未完全面向 Loona，后续才转向 Loona。

目标：P0 场景必须稳定唤醒，P1 场景理应可用，P2 场景可阶段性放宽。

### 11.3 多模态信号冲突

典型来源：

- 有人声但用户未看向 Loona。
- 用户看向 Loona 但没有同步唇动。
- 声源方向与视觉目标不一致。
- 多人同时说话，目标用户不清晰。

目标：系统应优先避免误唤醒，在高置信度条件下触发唤醒；低置信度场景可保持未唤醒或进入观察状态。

## 12. 非功能需求

### 12.1 性能要求

1. 系统需保持持续监听能力。
2. 从用户开始对 Loona 说话到触发唤醒事件，应尽量减少感知延迟。
3. 多模态信号融合判断不应造成明显卡顿。
4. 唤醒事件应包含足够的调试信息，便于后续分析误唤醒和唤不醒。

### 12.2 稳定性要求

1. 持续监听和多模态判断链路需稳定运行。
2. 环境噪声、多人说话、弱朝向等复杂场景下不应导致状态频繁抖动。
3. 系统应支持对唤醒判断结果进行日志记录和回放分析。

## 13. 验收标准

### 13.1 基础验收

满足以下条件可视为本阶段基础验收通过：

1. 系统可稳定识别 P0 应唤醒场景，并触发唤醒事件。
2. 系统可稳定拒绝 P0 不应唤醒场景，不频繁误触发。
3. 单人办公短句场景可稳定触发唤醒。
4. 多人场景中，明确面向 Loona 的目标用户输入可触发唤醒。
5. 背景人声、电视播报、视频音频不应高频误触发。
6. 唤醒事件应包含判断置信度和主要触发依据。

### 13.2 当前阶段优先验收重点

1. P0 应唤醒场景成功率。
2. P0 不应唤醒场景误唤醒率。
3. 多人场景下目标用户判断能力。
4. 短句场景唤醒成功率。
5. 背景人声和电脑外放场景的误触发控制。

## 14. 风险与待补充项

### 14.1 主要风险

1. 多模态意图判断若过于激进，可能带来更多误唤醒。
2. 多模态意图判断若过于保守，可能造成用户明确发起交互时唤不醒。
3. 短句场景信号少，容易出现漏唤醒。
4. 多人场景中目标用户识别困难，可能同时放大误唤醒和唤不醒问题。
5. 声源、视线、唇动等信号不同步时，可能造成判断不稳定。

### 14.2 后续实现方案需补充

1. 多模态无唤醒词判断的触发条件与边界规则。
2. 声源、朝向、视线、唇动、距离、上下文状态的融合策略。
3. 唤醒置信度计算方式和阈值策略。
4. 误唤醒过滤规则与误判控制指标。
5. 多人场景下目标用户选择策略。
6. P0 / P1 / P2 场景测试集与验收方法。

## 15. 建议后续讨论问题

为了进入实现方案阶段，建议优先确认以下问题：

1. 当前版本是否以 P0 场景作为上线阻塞标准，P1 作为体验优化标准，P2 作为观察项。
2. 无唤醒词判断是否仅在用户处于 Loona 有效感知距离内时开启。
3. 朝向、视线、唇动、声源、距离这些信号中，哪些是当前设备侧已经稳定可用的输入。
4. 唤醒事件是否需要输出目标用户 ID、置信度和触发依据。
5. 对误唤醒率、唤醒成功率、短句唤醒成功率是否已有量化目标。
